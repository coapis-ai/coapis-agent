## 一、问题背景

### 1.1 当前问题

**技术导向 vs 场景导向**

| 维度 | 当前状态 | 目标状态 |
|------|---------|---------|
| **菜单命名** | Skills、Tools、MCP | 会议纪要、工作报告、合同起草 |
| **功能入口** | 用户不知道能做什么 | 用户直接看到业务功能 |
| **学习成本** | 需要理解技术概念 | 零学习成本，开箱即用 |
| **使用效率** | 多次点击才能找到功能 | 一键直达业务场景 |

**具体问题**：

```
用户视角的问题：

1. 打开系统，看到的是"Skills"、"Tools"、"MCP"
   → 不知道这些是什么，不知道能做什么

2. 想做"会议纪要"，不知道从哪里开始
   → 是Chat？是Skills？还是Tools？

3. 进入Chat，只是一个空白对话框
   → 不知道怎么用，缺少引导

4. 技术配置和业务功能混在一起
   → 普通用户看到很多用不到的功能
```

### 1.2 设计目标

| 目标 | 描述 | 成功标准 |
|------|------|---------|
| **降低认知负担** | 用户打开系统就知道能做什么 | 新用户3秒内找到目标功能 |
| **清晰的角色区分** | 不同角色看到不同的功能 | 普通用户不看到管理功能 |
| **场景化入口** | 功能按业务场景组织 | 用户按场景找到功能，而非技术分类 |
| **深度场景代入** | 进入功能后有完整的引导和工具 | 用户无需思考"下一步做什么" |

## 二、用户角色分析

### 2.1 角色定义

系统支持三种用户角色：

| 角色 | 英文标识 | 典型用户 | 主要需求 | 权限范围 |
|------|---------|---------|---------|---------|
| **普通用户** | `user` | 业务人员、办事员 | 日常办公、专业业务 | 业务功能 |
| **团队管理员** | `team_admin` | 部门经理、项目经理 | 团队管理、统计监控 | 业务功能 + 团队管理 |
| **领导** | `leader` | 单位领导、分管领导 | 决策支持、信息查阅 | 业务功能 + 报表查看 |

### 2.2 角色需求矩阵

#### 普通用户需求

| 需求类别 | 具体需求 | 优先级 |
|---------|---------|--------|
| **通用办公** | 会议纪要、工作报告、邮件撰写、通知公告 | ⭐⭐⭐⭐⭐ |
| **专业业务** | 政策解读、合同起草、数据分析等 | ⭐⭐⭐⭐ |
| **个人管理** | 我的工作、历史记录、收藏夹 | ⭐⭐⭐⭐ |
| **设置** | 个人信息、偏好设置、安全设置 | ⭐⭐⭐ |

#### 团队管理员需求

| 需求类别 | 具体需求 | 优先级 |
|---------|---------|--------|
| **基础需求** | 继承普通用户所有需求 | ⭐⭐⭐⭐⭐ |
| **团队管理** | 成员管理、权限分配、角色配置 | ⭐⭐⭐⭐⭐ |
| **统计监控** | 使用统计、性能监控、异常告警 | ⭐⭐⭐⭐ |
| **配置管理** | 智能体配置、模型选择、工具配置 | ⭐⭐⭐ |

#### 领导需求

| 需求类别 | 具体需求 | 优先级 |
|---------|---------|--------|
| **基础需求** | 继承普通用户所有需求 | ⭐⭐⭐⭐⭐ |
| **决策支持** | 数据报表、统计分析、趋势预测 | ⭐⭐⭐⭐⭐ |
| **信息查阅** | 团队工作进展、重要事项提醒 | ⭐⭐⭐⭐ |
| **快捷访问** | 常用功能快捷入口、个人收藏 | ⭐⭐⭐⭐ |

### 2.3 不需要的功能

| 角色 | 不需要的功能 | 原因 |
|------|-------------|------|
| **普通用户** | Channels、Sessions、MCP、Models、Security、Token Usage、Backups、Debug、SSO、Admin | 这些是技术配置和管理功能 |
| **团队管理员** | SSO、Admin（部分）、Debug | 这些是系统级配置 |
| **领导** | 所有技术配置功能 | 领导只关注业务和决策 |

### 2.4 角色权限实现方案

#### 方案A：基于权限码的动态菜单

```typescript
// 菜单项配置
const menuItems = [
  {
    key: 'meeting-minutes',
    label: '会议纪要',
    permissions: ['user', 'team_admin', 'leader'], // 所有角色可见
    category: 'business'
  },
  {
    key: 'team-management',
    label: '团队管理',
    permissions: ['team_admin'], // 只有团队管理员可见
    category: 'management'
  },
  {
    key: 'models',
    label: '模型管理',
    permissions: ['admin'], // 只有系统管理员可见
    category: 'settings'
  }
]
```

#### 方案B：基于用户角色的视图切换

```typescript
// 根据用户角色显示不同的菜单视图
const getMenuByRole = (role: UserRole) => {
  switch (role) {
    case 'user':
      return businessMenuItems; // 只显示业务功能
    case 'team_admin':
      return [...businessMenuItems, ...managementMenuItems];
    case 'leader':
      return [...businessMenuItems, ...dashboardMenuItems];
    default:
      return businessMenuItems;
  }
}
```

## 三、界面布局设计

### 3.1 顶部Tab双视图方案

**核心设计**：在顶部栏分为两个Tab，区分业务场景和技术场景

```
┌─────────────────────────────────────────────────────────────┐
│  [我的工作]  [我的管理]                    [用户信息] [设置] │  ← 顶部栏
├─────────────────────────────────────────────────────────────┤
│                                                               │
│                      内容主窗口                                │
│                                                               │
│  （根据选中的Tab显示不同的内容）                               │
│                                                               │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Tab命名方案

| 方案 | "我的工作" | "我的管理" | 优点 | 缺点 |
|------|-----------|-----------|------|------|
| **方案1** | 我的工作 | 我的管理 | 直观，职责清晰 | "管理"一词可能让领导困惑 |
| **方案2** | 工作台 | 控制台 | 专业化，角色清晰 | 可能不够亲切 |
| **方案3** | 业务中心 | 管理中心 | 清晰区分业务和管理 | 名称较长 |
| **方案4** | 我的应用 | 系统设置 | 用户熟悉的概念 | "应用"可能被误解 |
| **方案5** | 工作 | 配置 | 简洁，通用 | 可能过于简单 |

**推荐方案**：方案2「工作台」+「控制台」
- 工作台：业务人员的工作场所，场景化功能入口
- 控制台：管理员的技术配置界面，专业化命名

## 四、业务功能分类

### 4.1 分类原则

**用户视角的分类**：按业务场景分类，而非技术分类

```
工作台功能分类：

1. 快捷功能（高频、通用）
   └── 用户最常用的功能，固定显示在最上方

2. 业务分类（领域导向）
   ├── 通用办公（所有用户都需要的）
   ├── 政府业务（政府用户）
   ├── 金融业务（金融用户）
   ├── 医疗业务（医疗用户）
   ├── 法律业务（法律用户）
   ├── 工程业务（工程用户）
   └── 教育业务（教育用户）

3. 最近使用（个性化推荐）
   └── 用户最近使用的功能，快速访问

4. 我的收藏（用户自定义）
   └── 用户收藏的常用功能
```

### 4.2 功能与技能的映射关系

**核心概念**：
- **业务功能** = 用户可见的功能入口（如"会议纪要"）
- **技能** = AI执行任务的能力单元（如"音频转写"、"内容提取"）
- **关系** = 一个业务功能可能组合多个技能

#### 示例：会议纪要功能

```
【会议纪要】业务功能
    │
    ├── 技能1：音频转写（gen_audio_transcribe）
    │   └── 输入：会议录音 → 输出：转写文本
    │
    ├── 技能2：内容提取（gen_content_extract）
    │   └── 输入：转写文本 → 输出：关键信息
    │
    ├── 技能3：格式整理（gen_format_organize）
    │   └── 输入：关键信息 → 输出：结构化内容
    │
    ├── 技能4：模板渲染（gen_template_render）
    │   └── 输入：结构化内容 → 输出：会议纪要文档
    │
    └── 技能5：任务识别（gen_task_identify）
        └── 输入：会议纪要 → 输出：待办事项列表
```

### 4.4 功能发现机制

用户如何发现和找到功能？

```
功能发现路径：

1. 【快捷功能区】
   └── 展示8-12个高频功能，用户一目了然

2. 【业务分类区】
   └── 按领域展开，用户按场景浏览

3. 【搜索功能】
   └── 用户搜索"会议"、"合同"等关键词

4. 【最近使用】
   └── 用户快速访问最近使用的功能

5. 【智能推荐】
   └── 根据用户角色、历史使用、时间等推荐功能
   └── 例如：周一早上推荐"周报"，会议后推荐"会议纪要"
```

```typescript
// 快捷卡片排序算法
const sortQuickCards = (cards: Card[], userHistory: Usage[]) => {
  return cards.sort((a, b) => {
    // 1. 用户历史使用频率
    const freqA = getUsageFrequency(a.id, userHistory);
    const freqB = getUsageFrequency(b.id, userHistory);
    
    // 2. 角色相关性权重
    const roleWeightA = getRoleRelevance(a.id, userRole);
    const roleWeightB = getRoleRelevance(b.id, userRole);
    
    // 3. 时间相关性（如周一早上推荐周报）
    const timeWeightA = getTimeRelevance(a.id);
    const timeWeightB = getTimeRelevance(b.id);
    
    // 综合评分
    const scoreA = freqA * 0.5 + roleWeightA * 0.3 + timeWeightA * 0.2;
    const scoreB = freqB * 0.5 + roleWeightB * 0.3 + timeWeightB * 0.2;
    
    return scoreB - scoreA;
  });
};
```

## 六、场景代入机制

### 6.1 场景代入的定义

**场景代入**：用户点击功能卡片后，系统自动加载该业务场景所需的所有元素，让用户"无感"进入工作状态。

**代入元素**：

| 元素 | 说明 | 示例（会议纪要） |
|------|------|-----------------|
| **对话角色** | AI扮演的角色 | "会议纪要助手" |
| **系统提示词** | AI的行为指导 | "你是会议纪要专家，擅长..." |
| **欢迎消息** | 用户进入后的第一句话 | "您好！我可以帮您生成会议纪要..." |
| **输入引导** | 指导用户如何开始 | 上传录音、输入笔记、选择模板 |
| **预设工具** | 启用该场景需要的工具 | 音频转写、文档解析、模板渲染 |
| **输出模板** | 预设输出格式 | 会议纪要模板（标题、时间、参会人、议程、决议...） |
| **工作流程** | 多步骤引导 | 步骤1→步骤2→步骤3→完成 |
| **示例内容** | 帮助用户理解 | 点击查看示例会议纪要 |

### 8.3 API设计

#### 业务功能API

```python
# server/api/business_functions.py

@router.get("/functions")
async def list_functions(
    category: Optional[str] = None,
    user_role: str = Depends(get_current_user_role)
):
    """获取业务功能列表"""
    # 根据用户角色过滤功能
    functions = await db.query(
        "SELECT * FROM business_functions WHERE allowed_roles @> :role",
        {"role": json.dumps([user_role])}
    )
    return functions

@router.get("/functions/{function_id}")
async def get_function_detail(function_id: str):
    """获取功能详情（包含场景配置）"""
    function = await db.get(
        "SELECT * FROM business_functions WHERE id = :id",
        {"id": function_id}
    )
    return function

@router.post("/functions/{function_id}/use")
async def record_function_usage(
    function_id: str,
    session_id: str,
    user_id: str = Depends(get_current_user_id)
):
    """记录功能使用"""
    await db.execute(
        "INSERT INTO user_function_usage (user_id, function_id, session_id) VALUES (:user_id, :function_id, :session_id)",
        {"user_id": user_id, "function_id": function_id, "session_id": session_id}
    )
    return {"success": True}

@router.get("/functions/recent")
async def get_recent_functions(
    user_id: str = Depends(get_current_user_id),
    limit: int = 10
):
    """获取用户最近使用的功能"""
    recent = await db.query(
        """
        SELECT f.*, MAX(u.used_at) as last_used
        FROM business_functions f
        JOIN user_function_usage u ON f.id = u.function_id
        WHERE u.user_id = :user_id
        GROUP BY f.id
        ORDER BY last_used DESC
        LIMIT :limit
        """,
        {"user_id": user_id, "limit": limit}
    )
    return recent
```

## 十、总结与建议

### 10.1 核心设计原则

| 原则 | 说明 |
|------|------|
| **用户视角** | 从用户能做什么出发，而非系统能提供什么 |
| **场景导向** | 功能按业务场景组织，而非技术分类 |
| **角色区分** | 不同角色看到不同的功能和界面 |
| **深度代入** | 用户进入功能后无需思考，有完整引导 |
| **渐进增强** | 不破坏现有功能，逐步迁移 |

### 10.2 关键成功因素

| 因素 | 说明 |
|------|------|
| **功能设计** | 每个业务功能都需要精心设计场景代入流程 |
| **用户反馈** | 及早获取用户反馈，迭代优化 |
| **性能优化** | 场景代入不应增加加载时间 |
| **文档完善** | 每个功能都有清晰的配置和使用文档 |
