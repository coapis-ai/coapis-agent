## 架构概览

CoApis 记忆系统采用纯文件存储 + LLM 增强的方案，不依赖任何向量数据库或外部服务。

```
┌─────────────────────────────────────────────────────┐
│                    记忆系统架构                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │ memory_search │  │  summarize   │  │   dream    │ │
│  │  关键词匹配   │  │  压缩时提取  │  │  定期整理  │ │
│  │  + LLM 重排  │  │  + LLM 分析  │  │  + LLM 合并│ │
│  └──────┬───────┘  └──────┬───────┘  └──────┬─────┘ │
│         │                 │                 │       │
│         └─────────────────┼─────────────────┘       │
│                           │                         │
│                    ┌──────▼──────┐                  │
│                    │  MEMORY.md  │  ← 长期记忆       │
│                    │  memory/*.md│  ← 每日笔记       │
│                    └─────────────┘                  │
│                                                     │
│  ┌─────────────────────────────────────────────┐   │
│  │           三层 Prompt 继承                    │   │
│  │  global < agent < user (SectionMerger)      │   │
│  └─────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

### 核心特性

| 特性 | 说明 |
|------|------|
| **零外部依赖** | 不需要 reme-ai、chromadb 或任何向量数据库 |
| **纯文件存储** | MEMORY.md + memory/*.md，人类可读可编辑 |
| **LLM 增强** | 关键词搜索 + LLM 语义重排，提升检索质量 |
| **自动提取** | 上下文压缩时自动提取高价值信息 |
| **定期整理** | Dream 机制定期合并每日笔记到长期记忆 |
| **用户隔离** | 每个用户独立的工作区和记忆文件 |

## 核心组件

### 1. ReMeLightMemoryManager

> 文件: `server/coapis/agents/memory/reme_light_memory_manager.py`

尽管类名保留了 `ReMeLightMemoryManager`（兼容现有配置），但已完全移除 reme-ai 依赖，实现为纯文件记忆管理器。

```python
@memory_registry.register("remelight")
class ReMeLightMemoryManager(BaseMemoryManager):
    """File-based memory manager with LLM-enhanced search."""
```

#### 核心方法

| 方法 | 触发时机 | 功能 |
|------|----------|------|
| `memory_search()` | Agent 调用工具 / retrieve 自动触发 | 关键词匹配 + LLM 重排 |
| `summarize()` | 上下文压缩时 | 从被压缩消息提取高价值信息 |
| `dream()` | Dream cron 触发（默认每晚 23:00） | 合并每日笔记 → MEMORY.md |
| `retrieve()` | Agent 回复前（自动） | 预取相关记忆注入上下文 |

### 2. 上下文压缩管理器

> 文件: `server/coapis/agents/context/light_context_manager.py`

当 token 数超过阈值时触发压缩，压缩时自动调用 `memory_manager.add_summarize_task()` 提取记忆。

### 3. Dream 调度器

> 文件: `server/coapis/app/crons/manager.py`

通过 cron 表达式定期触发 Dream 任务，默认每晚 23:00 执行。

## 功能详解

### 1. memory_search — 语义增强搜索

**流程**:

```
用户查询
  │
  ▼
Phase 1: 关键词匹配
  │ 收集 MEMORY.md + memory/*.md
  │ 分块（500字符，50字符重叠）
  │ 计算关键词重叠分数
  │ 筛选 ≥ min_score 的候选
  │
  ▼
Phase 2: LLM 语义重排（候选 > max_results 时）
  │ 将候选传给 LLM
  │ LLM 返回按相关性排序的索引
  │
  ▼
返回 ToolResponse（含文件路径、行号、内容摘要）
```

**关键词评分算法**:

```python
def _keyword_score(query, text):
    query_tokens = set(re.findall(r'[\w\u4e00-\u9fff]+', query.lower()))
    matched = sum(1 for t in query_tokens if t in text.lower())
    return matched / len(query_tokens)  # 0.0 ~ 1.0
```

**LLM 重排 Prompt**:
```
Query: {query}

Candidates:
[0] {preview_0}
[1] {preview_1}
...

Rank the candidates by relevance to the query.
Return ONLY a JSON array of indices in descending relevance.
```

### 2. summarize — 压缩时自动提取

**触发条件**: 上下文压缩时（`summarize_when_compact=True`）

**流程**:

```
上下文压缩触发
  │
  ▼
收集被压缩的消息（最近 20 条）
  │
  ▼
LLM 提取高价值信息
  │ - 用户偏好/习惯
  │ - 重要决策/结论
  │ - 关键事实/数据
  │ - 用户纠正的错误
  │ - 重要待办/承诺
  │
  ▼
去重检查（与 MEMORY.md 已有内容对比）
  │
  ▼
追加到 MEMORY.md
```

**提取 Prompt（中文）**:
```
从以下对话中提取值得长期记忆的信息。只提取以下类型：
1. 用户明确的偏好或习惯
2. 重要的决策或结论
3. 关键的事实或数据
4. 用户纠正过的错误认知
5. 重要的待办事项或承诺

不要提取：闲聊、一般性问答、临时性信息。

如果有值得记忆的内容，输出简洁的要点列表（每条一行，以 - 开头）。
如果没有值得记忆的内容，输出：无
```

### 3. dream — 定期记忆整理

**触发方式**: Cron 定时任务（默认 `0 23 * * *`，每晚 23:00）

**流程**:

```
Cron 触发
  │
  ▼
收集近 3 天的 memory/*.md 文件
  │
  ▼
读取现有 MEMORY.md
  │
  ▼
备份 MEMORY.md → backup/memory_backup_{timestamp}.md
  │
  ▼
LLM 合并去重
  │ - 保留现有有价值内容
  │ - 提取每日笔记新信息
  │ - 合并重复，保留最新
  │ - 删除过时内容
  │
  ▼
覆盖写入 MEMORY.md
```

**LLM 整理 Prompt（中文）**:
```
你是记忆整理专家。请将以下每日笔记与现有长期记忆合并，生成更新后的长期记忆。

规则：
1. 保留现有长期记忆中有价值的内容
2. 从每日笔记中提取新的有价值信息（偏好、决策、事实、待办）
3. 去重：合并重复信息，保留最新版本
4. 删除过时或不再相关的内容
5. 保持简洁，每条记忆不超过2行
6. 按主题分类组织
```

### 4. retrieve — 自动记忆预取

**触发时机**: Agent 回复前自动调用

**流程**:

```
Agent 收到用户消息
  │
  ▼
提取消息文本（最近 100 字符）
  │
  ▼
调用 memory_search(query, max_results, min_score)
  │
  ▼
构造 ToolUse + ToolResult 消息对
  │
  ▼
注入到消息上下文中
  │
  ▼
Agent 回复时可引用记忆内容
```

## Prompt 继承机制

### 三层合并

CoApis 使用 `SectionMerger` 按 `##` section 合并三层 Prompt：

```
优先级: user (2) > agent (1) > global (0)

┌─────────────────┐
│  global 层      │  system/templates/*.md
│  (优先级 0)     │
├─────────────────┤
│  agent 层       │  agents/{agent_id}/*.md
│  (优先级 1)     │
├─────────────────┤
│  user 层        │  workspaces/{user_id}/*.md
│  (优先级 2)     │
└─────────────────┘
         │
         ▼
  SectionMerger 合并
         │
         ▼
  最终系统 Prompt
```

### 合并规则

1. **按 section 合并**: 同名 `## section` 按优先级覆盖
2. **高优先级覆盖低优先级**: user 层的 `## 记忆` 覆盖 agent 层的 `## 记忆`
3. **独有 section 保留**: 各层独有的 section 全部保留
4. **MEMORY.md 不注入系统 Prompt**: 通过 memory_search/retrieve 工具动态检索

### 关键文件

| 文件 | 层级 | 说明 |
|------|------|------|
| `SOUL.md` | global → agent → user | 人格设定 |
| `AGENTS.md` | global → agent → user | 行为规则 |
| `PROFILE.md` | global → agent → user | 用户资料 |
| `MEMORY.md` | user only | 长期记忆（工具检索） |
| `BOOTSTRAP.md` | agent only | 启动引导 |

## 故障排查

### 问题 1: memory_search 返回空结果

**可能原因**:
- MEMORY.md 和 memory/ 目录为空
- 查询词与记忆内容无重叠
- min_score 设置过高

**排查步骤**:
```bash
# 检查记忆文件是否存在
ls -la workspaces/{user_id}/MEMORY.md
ls -la workspaces/{user_id}/memory/

# 检查文件内容
cat workspaces/{user_id}/MEMORY.md

# 降低 min_score 测试
memory_search(query="xxx", min_score=0.01)
```

### 问题 2: Dream 未执行

**可能原因**:
- dream_cron 配置为空
- CronManager 未启动
- 无 memory/*.md 文件

**排查步骤**:
```bash
# 检查 Dream cron 配置
grep "dream_cron" workspaces/{user_id}/agent.json

# 检查服务器日志
docker logs coapis-server-dev 2>&1 | grep -i dream

# 检查 CronManager 状态
docker logs coapis-server-dev 2>&1 | grep "CronManager"
```

### 问题 3: 压缩时未提取记忆

**可能原因**:
- `summarize_when_compact` 设为 false
- 上下文未达到压缩阈值
- LLM 调用失败

**排查步骤**:
```bash
# 检查配置
grep "summarize_when_compact" workspaces/{user_id}/agent.json

# 检查日志
docker logs coapis-server-dev 2>&1 | grep -i "MemoryExtract\|summarize\|compact"
```

### 问题 4: LLM 重排失败

**可能原因**:
- LLM 服务不可用
- 网络问题
- Token 超限

**排查步骤**:
```bash
# 检查 LLM 服务
curl -s http://localhost:4308/api/models/available | python3 -m json.tool

# 检查日志
docker logs coapis-server-dev 2>&1 | grep "LLM rerank"
```

## 进化系统审计与自动化审核

> **版本**: v0.8.34  
> **更新日期**: 2026-06-26

### 架构概览

进化系统（EvolutionEngine / CrossAgentEvolution / KnowledgeFlow）始终开启，无需总开关。所有进化变更通过**审计日志**记录，并根据**风险分级**决定审核方式。

```
┌─────────────────────────────────────────────────────────┐
│                   进化系统审核流程                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  经验提取 → B桶积累 → AI评审(阈值触发)                     │
│                        ↓                                │
│              ┌─────────┴─────────┐                      │
│              │   风险分级判断      │                      │
│              └─────────┬─────────┘                      │
│         ┌──────────────┼──────────────┐                 │
│         ↓              ↓              ↓                 │
│    L0 自动通过     L1 LLM审核     L2 人工审核              │
│   (可泛化+高置信)  (可泛化+中置信) (不可泛化/低置信)         │
│         ↓              ↓              ↓                 │
│    直接晋升基础层   LLM批准→晋升   等管理员确认              │
│         ↓          LLM拒绝→归档       ↓                  │
│    审计日志记录      审计日志记录   审计日志记录              │
│         ↓              ↓              ↓                 │
│       回滚支持       回滚支持       暂不支持                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 风险分级规则

| 级别 | 条件 | 审核方式 | 行为 |
|------|------|----------|------|
| **L0** | `is_generalizable=True` 且 `confidence ≥ AUTO_APPROVE_CONFIDENCE` | 自动 | 直接晋升到基础层，记录审计日志 |
| **L1** | `is_generalizable=True` 且 `confidence ≥ min_confidence` | LLM | LLM二次审核，批准则晋升，拒绝则归档 |
| **L2** | `is_generalizable=False` 或 `confidence < min_confidence` | 人工 | 进入A桶等待管理员确认 |

### 环境变量

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `COAPIS_AUTO_APPROVE_CONFIDENCE` | `0.8` | L0自动通过最低置信度阈值 |
| `COAPIS_SKILL_AUTO_REVIEW_THRESHOLD` | `0.5` | 全局技能优化触发阈值（composite_score低于此值触发优化建议） |

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/evolution/audit-log` | 查询审计日志（支持 `change_type`/`target_type`/`risk_level`/`review_method` 筛选） |
| `GET` | `/api/evolution/audit-log/stats` | 统计概览（总数/按风险/按审核方式/按决策） |
| `GET` | `/api/evolution/audit-log/{entry_id}` | 查询单条记录 |
| `POST` | `/api/evolution/audit-log/{entry_id}/rollback` | 执行回滚（仅 `rollback_available=true` 的记录可用） |

### 回滚机制

回滚操作从审计日志中读取 `content_before` 恢复目标文件，仅支持 `memory` 和 `skill` 类型的回滚。执行回滚后：
1. 恢复目标文件到 `content_before` 内容
2. 写入新的 `rollback` 类型审计记录
3. 标记原记录 `rolled_back=true`

### 全局技能效能监控

当全局技能的 `composite_score < SKILL_AUTO_REVIEW_THRESHOLD` 时，系统通过 KnowledgeFlow 触发优化建议（L1 LLM审核），记录审计日志。五维效能指标：

| 指标 | 来源 | 说明 |
|------|------|------|
| `precision` | `tool_usage.jsonl` 成功/失败计数 | 触发准确性 |
| `reliability` | 任务成功率 | 任务完成率 |
| `effectiveness` | 工具调用效率 | 资源利用效率 |
| `satisfaction` | 主动反馈评分 | 用户满意度 |
| `robustness` | 错误恢复次数 | 异常恢复能力 |
