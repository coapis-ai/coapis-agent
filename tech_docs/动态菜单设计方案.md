# 动态菜单设计方案

## 当前状态

**菜单配置位置**：
- 前端硬编码：`client/src/config/menuConfig.tsx`
- 二级菜单：工作场景从标签系统动态加载 ✅

**权限系统现状**：
- 权限模块：`server/coapis/data/packs/base/auth/permissions.json`
- 前端过滤：`Sidebar.tsx` 根据 `allowedModules` 过滤菜单
- 映射配置：`client/src/config/permissionMapping.ts`

**问题**：
- 一级菜单硬编码，修改需要重新构建前端
- 菜单顺序、图标、名称等无法动态调整
- 新增菜单项需要修改代码

---

## 方案对比

### 方案1：菜单配置存数据库 ⭐⭐

**优点**：
- ✅ 完全动态，可在UI中配置
- ✅ 支持菜单-权限关联
- ✅ 无需重启服务

**缺点**：
- ❌ 增加权限管理复杂性
- ❌ 需要新增表：`menu_items`, `menu_permissions`
- ❌ 需要管理界面

**数据结构**：
```sql
CREATE TABLE menu_items (
  id VARCHAR(50) PRIMARY KEY,
  parent_id VARCHAR(50),
  label_key VARCHAR(100),
  icon VARCHAR(50),
  path VARCHAR(200),
  sort_order INT,
  is_active BOOLEAN
);

CREATE TABLE menu_permissions (
  menu_id VARCHAR(50),
  permission_module VARCHAR(50),
  PRIMARY KEY (menu_id, permission_module)
);
```

**适用场景**：
- 多租户系统
- 需要细粒度权限控制
- 有专门的运维团队

### 方案2：菜单配置存JSON文件 ⭐⭐⭐

**优点**：
- ✅ 简单，无需数据库
- ✅ 人工可编辑
- ✅ 版本控制友好

**缺点**：
- ❌ 修改需要重启服务
- ❌ 无管理界面

**数据结构**：
```json
{
  "version": "1.0",
  "items": [
    {
      "key": "chat",
      "labelKey": "nav.chat",
      "icon": "MessageOutlined",
      "path": "/chat",
      "permission": "chat",
      "sortOrder": 1,
      "isActive": true
    },
    {
      "key": "workbench",
      "labelKey": "nav.workbench",
      "icon": "AppstoreOutlined",
      "path": "/workbench",
      "permission": null,
      "sortOrder": 2,
      "childrenSource": "api:/api/scenes/categories/grouped"
    }
  ]
}
```

**实现要点**：
- 后端读取配置文件，通过API返回菜单数据
- 前端从API加载菜单配置
- 支持二级菜单数据源（API或静态数据）

### 方案3：菜单配置存YAML文件 + 热重载 ⭐⭐⭐⭐

**优点**：
- ✅ 简单，人工可编辑
- ✅ 支持热重载（无需重启）
- ✅ 注释友好，易于维护

**缺点**：
- ❌ 无管理界面
- ❌ 需要文件监听机制

**数据结构**：
```yaml
# 菜单配置 v1.0
version: "1.0"

# 一级菜单
items:
  - key: chat
    labelKey: nav.chat
    icon: MessageOutlined
    path: /chat
    permission: chat
    sortOrder: 1
    
  - key: workbench
    labelKey: nav.workbench
    icon: AppstoreOutlined
    path: /workbench
    permission: null  # 无权限限制
    sortOrder: 2
    # 二级菜单数据源
    childrenSource:
      type: api
      url: /api/scenes/categories/grouped
      transform: dimensions.nature.categories  # 数据转换路径
    
  - key: myspace
    labelKey: nav.myspace
    icon: FolderOutlined
    path: /workspace/myspace
    permission: myspace
    sortOrder: 3
    
  - key: settings
    labelKey: nav.settings
    icon: SettingOutlined
    path: /settings
    permission: settings
    sortOrder: 4

# 企业版扩展菜单
enterpriseExtensions:
  - key: knowledge
    labelKey: nav.knowledge
    icon: BookOutlined
    path: /knowledge
    permission: knowledge
    sortOrder: 3.5  # 插入到 myspace 和 settings 之间
```

**实现要点**：
- 使用 `watchdog` 监听文件变化
- 文件变化时重新加载配置
- 通过 WebSocket 推送更新到前端

### 方案4：标签系统扩展 ⭐⭐⭐⭐⭐（推荐）

**优点**：
- ✅ 复用现有标签系统
- ✅ 无需新增配置系统
- ✅ 工作场景已验证可行
- ✅ 统一数据模型

**缺点**：
- ❌ 标签需要扩展字段
- ❌ 混合了不同用途的数据

**数据结构**：
```json
{
  "version": "4.0",
  "tags": [
    {
      "id": "menu-chat",
      "name": "聊天",
      "icon": "MessageOutlined",
      "type": "menu",
      "metadata": {
        "labelKey": "nav.chat",
        "path": "/chat",
        "permission": "chat",
        "sortOrder": 1,
        "isActive": true
      }
    },
    {
      "id": "menu-workbench",
      "name": "工作场景",
      "icon": "AppstoreOutlined",
      "type": "menu",
      "metadata": {
        "labelKey": "nav.workbench",
        "path": "/workbench",
        "permission": null,
        "sortOrder": 2,
        "childrenSource": {
          "type": "tag",
          "dimension": "nature",
          "categoryType": "category"

      }

```

**实现要点**：
- 扩展标签类型：新增 `type: "menu"`
- 使用 `metadata` 存储菜单配置
- 后端提供菜单API：`/api/menus`
- 前端从API加载菜单配置

## 推荐方案：标签系统扩展

### 为什么推荐？

1. **复用现有系统**：
   - 工作场景二级菜单已从标签系统加载
   - 验证了可行性
   - 无需新增配置文件或数据库表

2. **统一数据模型**：
   - 标签系统已有维度、分类体系
   - 菜单项可以看作一种特殊的标签
   - 降低系统复杂度

3. **易于维护**：
   - 一个文件管理所有标签和菜单
   - 版本控制友好
   - 人工可编辑

### 实施步骤

**阶段1：扩展标签模型**

```python
# server/coapis/models/tag.py
class Tag(BaseModel):
    id: str
    name: str
    icon: str
    type: str  # dimension, category, industry, frequency, menu
    dimension: Optional[str]  # 所属维度
    parent_id: Optional[str]  # 父标签ID
    metadata: Optional[dict]  # 扩展字段（菜单配置）
```

**阶段2：菜单API**

```python
# server/coapis/routers/menu.py
@router.get("/menus")
async def get_menus():
    """获取菜单配置"""
    tags = tag_service.get_tags_by_type("menu")
    menus = []
    for tag in tags:
        menu = {
            "key": tag.id,
            "label": tag.name,
            "icon": tag.icon,
            "path": tag.metadata.get("path"),
            "permission": tag.metadata.get("permission"),
            "sortOrder": tag.metadata.get("sortOrder", 0),
        }
        
        # 处理二级菜单数据源
        if "childrenSource" in tag.metadata:
            source = tag.metadata["childrenSource"]
            if source["type"] == "tag":
                menu["children"] = get_children_from_tags(source)
            elif source["type"] == "api":
                menu["childrenUrl"] = source["url"]
        
        menus.append(menu)
    
    return {"items": sorted(menus, key=lambda x: x["sortOrder"])}
```

**阶段3：前端动态加载**

```typescript
// client/src/layouts/Sidebar.tsx
const [menuItems, setMenuItems] = useState<MenuItem[]>([]);

useEffect(() => {
  fetch('/api/menus')
    .then(res => res.json())
    .then(data => {
      setMenuItems(data.items);
    });
}, []);

// 替换硬编码的 MAIN_MENU_ITEMS
const collapsedNavItems = menuItems.map(item => ({
  key: item.key,
  icon: getIconComponent(item.icon),
  path: item.path,
  label: t(item.labelKey, item.label),
}));
```

**阶段4：权限过滤**

```typescript
// 保持现有权限过滤逻辑
const filteredMenus = filterMenuItems(collapsedNavItems);
```

## 风险评估

### 方案1：数据库
- **风险**：权限关联复杂，容易出错
- **缓解**：简化设计，不实现菜单-权限关联

### 方案2：JSON
- **风险**：修改需要重启服务
- **缓解**：接受重启成本（配置变更频率低）

### 方案3：YAML+热重载
- **风险**：文件监听机制可能不稳定
- **缓解**：使用成熟的 `watchdog` 库

### 方案4：标签扩展
- **风险**：标签数据模型变得复杂
- **缓解**：使用 `metadata` 隔离菜单特定字段

---

## 决策建议

**推荐方案4：标签系统扩展** ⭐⭐⭐⭐⭐

**理由**：
1. ✅ 复用现有系统，无需新增配置系统
2. ✅ 工作场景已验证可行
3. ✅ 实施成本低，风险可控
4. ✅ 符合"简单优先"原则

**不增加权限复杂性的方法**：
- 菜单的 `permission` 字段指向现有权限模块
- 前端保持现有权限过滤逻辑
- 不实现菜单-权限独立关联
