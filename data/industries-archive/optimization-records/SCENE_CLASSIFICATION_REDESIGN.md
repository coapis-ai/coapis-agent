# 场景分类体系重新设计：多维度分类而非固定层级

> **分析日期**：2026-07-17
> **关键洞察**：场景分类不是固定的三级层级，而是多个平行的分类维度

---

## 一、问题重新诊断

### 1.1 我之前的错误理解

**错误理解**：分类是固定的三级层级
```
❌ 错误方案：
一级分类（功能分类）
├── 文档处理（二级分类）
│   ├── 会议纪要（三级实体）
│   ├── 公文起草（三级实体）
│   └── 工作报告（三级实体）
├── 审批服务（二级分类）
│   ├── 项目审批（三级实体）
│   └── ...
└── ...
```

**问题**：
- ❌ 把"文档处理"、"审批服务"当成了分类层级，而不是场景本身的性质
- ❌ 强制所有场景都放在固定的三级结构下
- ❌ 忽略了场景可以从多个维度进行分类的事实
- ❌ 一个场景只能属于一个分类，无法多维度展示

---

### 1.2 正确的理解

**场景的本质**：
- ✅ 场景是**扁平的实体**，不是层级结构
- ✅ 场景可以有**多个属性**（性质、领域、频率等）
- ✅ 分类是**基于属性聚合场景的方式**

**正确的分类思路**：

```
场景池（扁平实体）：
┌─────────────────────────────────┐
│ 会议纪要                         │
│ 性质：办公通用                    │
│ 领域：[自然资源, 生态环境, ...]   │
│ 频率：高频使用                    │
└─────────────────────────────────┘

分类维度（聚合方式）：

维度1：按性质分类
├── 办公通用
│   ├── 会议纪要 ← 从场景池聚合
│   ├── 公文起草 ← 从场景池聚合
│   └── 工作报告 ← 从场景池聚合
├── 审批服务
│   ├── 项目审批 ← 从场景池聚合
│   ├── 用地审批 ← 从场景池聚合
│   └── ...
└── 监管执法
    ├── 执法检查 ← 从场景池聚合
    └── ...

维度2：按领域分类
├── 自然资源
│   ├── 会议纪要 ← 从场景池聚合（领域适配版）
│   ├── 用地审批 ← 从场景池聚合
│   └── ...
├── 生态环境
│   ├── 会议纪要 ← 从场景池聚合（领域适配版）
│   ├── 环评审批 ← 从场景池聚合
│   └── ...
└── ...
```

**核心洞察**：
1. ✅ **场景是扁平的**，不是嵌套在层级中
2. ✅ **分类是多维度的**，可以有"按性质"、"按领域"、"按频率"等
3. ✅ **一个场景可以从属于多个分类**（会议纪要既属于"办公通用"，又属于"自然资源"领域）
4. ✅ **分类只是聚合方式**，不是场景的本质属性

## 二、分类维度的深入分析

### 2.1 维度1：按性质分类（场景性质）

**什么是性质分类**：
- 场景的**功能性质**，描述场景"做什么"
- 这是场景的**核心属性**，每个场景都有且只有一个性质

**性质分类列表**：

| 性质分类 | 含义 | 典型场景 | 数量预估 |
|---------|------|---------|---------|
| **办公通用** | 日常办公场景，所有领域通用 | 会议纪要、公文起草、工作报告 | 3个 |
| **审批服务** | 行政审批场景，流程导向 | 项目审批、用地审批、环评审批 | 8个 |
| **规划编制** | 规划方案编制，专业性强 | 空间规划、发展规划、专项规划 | 5个 |
| **监管执法** | 监督检查执法，合规导向 | 执法检查、安全检查、投诉处理 | 7个 |
| **数据分析** | 数据统计分析，决策支持 | 经济分析、环境监测、质量监测 | 5个 |
| **公共服务** | 公共服务场景，服务导向 | 咨询答复、信息公开、便民服务 | 4个 |
| **应急处置** | 突发事件应对，应急响应 | 应急响应、灾害预警、舆情处置 | 3个 |

**分类数量**：7个

**展示方式**：
```
按性质分类：
├── 办公通用（3个）
│   ├── 会议纪要
│   ├── 公文起草
│   └── 工作报告
├── 审批服务（8个）
│   ├── 项目审批
│   ├── 用地审批
│   └── ...
└── ...
```

**为什么性质分类应该是"一级直接展示"**：
- ✅ 这是场景的核心属性，用户最关心的"场景性质"
- ✅ 每个场景都有且只有一个性质，分类清晰
- ✅ 性质分类数量适中（7个），不会造成选择困难
- ✅ 用户选择性质后，直接看到该性质下的所有场景

---

### 2.2 维度2：按领域分类（业务领域）

**什么是领域分类**：
- 场景所服务的**业务领域**
- 一个场景可以支持多个领域（通用场景）
- 一个场景也可能只属于一个领域（专属场景）

**领域分类列表**：

| 领域分类 | 主管部门 | 典型场景 | 通用场景数 | 专属场景数 |
|---------|---------|---------|----------|----------|
| **自然资源** | 资规局 | 用地审批、不动产登记、矿山监管 | 5个 | 3个 |
| **生态环境** | 生态环境局 | 环评审批、环境监测、应急响应 | 5个 | 2个 |
| **农业农村** | 农业农村局 | 振兴规划、质量认证、技术推广 | 5个 | 3个 |
| **发展改革** | 发改委 | 规划编制、经济分析、价格监测 | 5个 | 2个 |
| **城乡建设** | 住建局 | 施工许可、质量安全、住房保障 | 5个 | 3个 |
| **教育管理** | 教育局 | 招生组织、教师培训、质量监测 | 5个 | 4个 |
| **林草湿荒** | 林业局 | 采伐审批、野生动物保护、湿地保护 | 5个 | 3个 |
| **文化旅游** | 文旅局 | 文物保护、非遗保护、旅游规划 | 5个 | 2个 |
| **卫生健康** | 卫健委 | 医疗机构审批、疾病预防、健康促进 | 5个 | 3个 |
| **综合执法** | 综合执法局 | 案件办理、执法检查、投诉处理 | 5个 | 1个 |

**分类数量**：10个

**展示方式**：
```
按领域分类：
├── 自然资源（8个场景）
│   ├── 会议纪要（领域适配版）
│   ├── 公文起草（领域适配版）
│   ├── 用地审批
│   ├── 不动产登记
│   └── ...
├── 生态环境（7个场景）
│   ├── 会议纪要（领域适配版）
│   ├── 环评审批
│   └── ...
└── ...
```

**关键特性**：
- ✅ **通用场景可以在多个领域展示**（如会议纪要出现在所有10个领域）
- ✅ **专属场景只在一个领域展示**（如不动产登记只在自然资源领域）
- ✅ **领域分类提供"我的工作"视角**，用户看到自己领域的所有相关场景

---

### 2.3 维度对比分析

| 维度 | 分类依据 | 分类数量 | 展示特点 | 用户价值 |
|------|---------|---------|---------|---------|
| **按性质分类** | 场景功能性质 | 7个 | 每个场景只属于一个性质 | "我做什么类型的工作" |
| **按领域分类** | 业务领域 | 10个 | 通用场景可属于多个领域 | "我所在领域有哪些工具" |
| **按频率分类** | 使用频率 | 3个 | 动态统计 | "我常用的工具有哪些" |

## 四、两种维度的对比分析

### 4.1 按性质分类 vs 按领域分类

| 维度 | 分类依据 | 分类数量 | 场景归属 | 用户价值 | 适用场景 |
|------|---------|---------|---------|---------|---------|

| **按领域分类** | 业务领域 | 10个 | 通用场景可属于多个领域 | "我所在领域有哪些工具" | 领域导向用户 |

### 4.2 两种维度的使用场景

**按性质分类适用场景**：
1. ✅ 新用户探索：用户不知道有哪些工具，按性质浏览
2. ✅ 跨领域工作：用户需要多种类型的工具，按性质查找
3. ✅ 功能导向：用户关心"做什么"，不关心"哪个领域"

**按领域分类适用场景**：
1. ✅ 专业用户：用户熟悉自己领域，直接找到领域相关工具
2. ✅ 领域工作：用户只需要自己领域的工具
3. ✅ 上下文适配：系统根据用户所在部门自动注入领域上下文

## 五、实施建议

### 5.1 界面设计要点

**要点1：维度切换器**
- 位置：顶部，醒目位置
- 形式：Tab 切换或下拉选择
- 默认：根据用户角色智能选择（部门用户默认"按领域"，综合用户默认"按性质"）

**要点2：一级分类菜单**
- 形式：左侧菜单或顶部 Tab
- 数量：每个维度不超过10个一级分类
- 信息：分类名称 + 图标 + 场景数量

**要点3：场景卡片网格**
- 形式：卡片网格，2-3列
- 信息：场景名称、描述、标签（高频、通用/专属）
- 交互：点击打开聊天窗口

---

### 5.2 数据结构实现

**数据表设计（关系型数据库）**：

```sql
-- 场景表
CREATE TABLE scenes (
  id VARCHAR(50) PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  description TEXT,
  category VARCHAR(50),  -- 性质分类
  frequency VARCHAR(20), -- 使用频率
  is_generic BOOLEAN,    -- 是否通用
  agent_id VARCHAR(50),  -- 关联智能体
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);

-- 场景-领域关联表（多对多）
CREATE TABLE scene_domains (
  scene_id VARCHAR(50),
  domain_id VARCHAR(50),
  PRIMARY KEY (scene_id, domain_id)
);

-- 领域表
CREATE TABLE domains (
  id VARCHAR(50) PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  icon VARCHAR(10),
  description TEXT,
  domain_context JSONB, -- 领域上下文配置
  created_at TIMESTAMP
);
```

**查询示例**：

```sql
-- 按性质分类查询场景
SELECT * FROM scenes WHERE category = 'office-common';

-- 按领域分类查询场景
SELECT s.* 
FROM scenes s
JOIN scene_domains sd ON s.id = sd.scene_id
WHERE sd.domain_id = 'natural-resources';

-- 查询通用场景
SELECT * FROM scenes WHERE is_generic = true;
```

## 六、总结

### 6.1 核心洞察

1. ✅ **场景是扁平的**，不是嵌套在层级中
2. ✅ **分类是多维度的**，可以有"按性质"、"按领域"等
3. ✅ **一个场景可以从属于多个分类**（通用场景出现在多个领域）
4. ✅ **分类只是聚合方式**，不是场景的本质属性
5. ✅ **性质分类应该是"一级直接展示"**，不需要再分层

### 6.2 关键决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **分类方式** | 多维度分类 | 一个场景可以从多个维度查找 |
| **维度数量** | 2-3个 | 性质、领域、频率（可选） |
| **展示方式** | 维度切换 + 扁平展示 | 简单直观，符合用户认知 |
| **一级分类数量** | 每个维度不超过10个 | 符合认知经济原则 |
